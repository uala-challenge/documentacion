Este proyecto está diseñado con una arquitectura orientada a eventos, utilizando AWS SNS y SQS para el procesamiento asíncrono de mensajes y una combinación de DynamoDB y Redis como bases de datos. Se estructura en varios microservicios que trabajan en conjunto para manejar la publicación de tweets, la actualización de líneas de tiempo y la gestión de seguidores.

Componentes de la Arquitectura

1. Publicación de Tweets - SNS & SQS

  • Cuando un usuario publica un tweet, este es enviado al SNS UalaChallenge.

  • SNS distribuye el mensaje a SQS Tweets, una cola de mensajes encargada de almacenar los eventos relacionados con la publicación de tweets.

2. Procesamiento de Tweets - event-processor

  • event-processor es un microservicio que consume mensajes de SQS Tweets y los procesa.

  • Su función principal es recibir los tweets publicados y actualizar la información en Redis.

  • También es responsable de propagar los tweets a los seguidores del usuario que lo publicó.

3. Servicio de Tweets - tweets-service

  • Expone las siguientes APIs:

    • GET /tweet/{tweet_id}/user/{user_id} → Obtiene un tweet específico de DynamoDB.

    • POST /tweet → Permite a los usuarios publicar un nuevo tweet, almacenándolo en DynamoDB y enviándolo a SNS.

  • Es el punto de entrada para la creación de tweets en la plataforma.

4. Bases de Datos - DynamoDB y Redis

  • DynamoDB:

    • Base de datos principal para almacenar tweets de forma persistente.

    • Se usa para garantizar alta disponibilidad y escalabilidad sin necesidad de administración de infraestructura.

  • Redis Uala:

    • Utilizado para manejar las líneas de tiempo de los usuarios con rapidez.

    • Almacena los tweets recientes en las timelines para mejorar el rendimiento de lectura.

5. Servicio de Línea de Tiempo - timeline-service

  • Expone las siguientes APIs:

    • GET /timeline/{user_id} → Obtiene la línea de tiempo del usuario desde Redis.

    • PATCH /timeline/{user_id} → Actualiza la línea de tiempo con nuevos tweets de los usuarios que sigue.

  • Funciona como un caché eficiente para recuperar rápidamente tweets relevantes.

6. Servicio de Gestión de Seguidores - users-service

  • Expone la API:

    • PATCH /follow/{user_id} → Permite a un usuario seguir a otro, almacenando esta relación en Redis.

  • Mantiene la lista de seguidores para cada usuario en Redis, lo que permite actualizaciones rápidas en la línea de tiempo.

Flujo de Datos

  1. Publicación de un Tweet:

    • Un usuario publica un tweet mediante tweets-service.

    • Se almacena en DynamoDB y se envía un evento a SNS.

    • SNS lo reenvía a SQS Tweets.

  2. Procesamiento del Tweet:

    • event-processor consume el mensaje de SQS y propaga el tweet a los seguidores del usuario en Redis.

    • La línea de tiempo de los seguidores se actualiza con este nuevo tweet.

  3. Consulta de Línea de Tiempo:

    • Un usuario consulta su timeline en timeline-service, que recupera los datos desde Redis.

    • Si el tweet no está en Redis, se puede recuperar desde DynamoDB a través de tweets-service.

  4. Seguimiento de Usuarios:

    • Un usuario sigue a otro mediante users-service, que almacena la relación en Redis.

    • Cuando el seguido publica un tweet, su tweet se distribuye automáticamente a los seguidores en timeline-service.

Posibles Beneficios de la Arquitectura

Escalabilidad: Uso de colas de mensajes con SNS y SQS permite manejar grandes volúmenes de tráfico sin afectar la latencia.
Baja latencia: Redis proporciona respuestas rápidas para la recuperación de la línea de tiempo.
Persistencia: DynamoDB almacena tweets de forma duradera.
Flexibilidad: La separación de servicios permite escalar de manera independiente cada componente.
Procesamiento Asíncrono: El uso de event-processor asegura que las actualizaciones no bloqueen las operaciones de los usuarios.

Enfoque de Desarrollo

Para este desafío, me enfoqué en diseñar una arquitectura escalable, eficiente y orientada a eventos, asegurando que el sistema pueda manejar grandes volúmenes de tráfico sin afectar el rendimiento. Para lograrlo, utilicé SNS y SQS como colas de mensajes para procesar eventos de manera asíncrona y optimicé la recuperación de datos con Redis y DynamoDB.

A nivel de desarrollo, decidí estructurar la aplicación bajo el enfoque de Package-Oriented Design y Clean Architecture, lo cual se ajusta perfectamente a Go, ya que permite organizar las funcionalidades en paquetes en lugar de clases, manteniendo el código modular, reutilizable y fácil de mantener.

Además, creé una librería centralizada para reducir la duplicidad de código y optimizar la gestión de dependencias en los distintos microservicios.
